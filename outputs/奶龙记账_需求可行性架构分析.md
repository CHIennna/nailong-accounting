# 奶龙记账：需求、可行性与架构分析文档

版本：v0.1  
日期：2026-06-08  
目标平台：Android 移动端  
项目定位：轻量、可爱、智能的个人记账 App

## 1. 项目概述

“奶龙记账”是一款面向 Android 手机用户的个人记账应用，核心目标是帮助用户快速记录日常收支、管理多个账本、控制预算，并通过 DeepSeek API 生成消费分析报告。

产品不追求一开始就做成复杂的财务管理系统，而是优先保证日常记账体验足够顺手、预算信息足够清晰、消费分析足够有用。

核心关键词：

- 快速记账
- 多账本管理
- 月度预算
- 分类预算
- 消费图表
- DeepSeek AI 消费分析
- 本地优先、隐私友好

## 2. 目标用户

主要用户包括：

- 学生：记录生活费、餐饮、购物、娱乐、学习支出。
- 上班族：记录工资、房租、通勤、餐饮、订阅服务等。
- 情侣或家庭用户：通过不同账本记录共同开销、旅行开销、家庭支出。
- 想控制消费但不想使用复杂财务软件的普通用户。

产品风格建议保持简单、可爱、轻量、低压力。用户打开应用后应该能快速理解当前财务状态，而不是被复杂报表和设置项打断。

## 3. 产品定位

奶龙记账的定位是：

> 一款带有 AI 消费分析能力的可爱型个人记账 App。

核心价值：

- 让用户快速记录每一笔收入和支出。
- 让用户清楚知道本月花了多少钱、还剩多少预算。
- 让用户通过图表理解消费结构。
- 让用户通过 AI 获得更容易理解的消费总结和节省建议。

## 4. 核心功能需求

### 4.1 首页概览

首页用于展示用户当前账本的整体财务状态。

需要展示：

- 当前账本
- 本月支出
- 本月收入
- 本月结余
- 预算剩余
- 最近账单
- 快速记账入口

首页的目标是让用户一打开 App 就能知道：

- 这个月花了多少
- 收入有多少
- 预算还剩多少
- 最近记录了哪些账单

### 4.2 快速记账

记账是项目最核心的功能。

支持类型：

- 支出
- 收入
- 转账

账单字段：

- 金额
- 分类
- 账户
- 所属账本
- 日期
- 备注

支出记账流程：

```text
点击记账
→ 默认选择支出
→ 输入金额
→ 选择分类
→ 选择账户
→ 选择账本
→ 保存
→ 返回首页并刷新统计
```

转账流程：

```text
选择转账
→ 输入金额
→ 选择转出账户
→ 选择转入账户
→ 保存
```

第一版中，转账建议只作为账户间资金流转记录，不计入收入和支出统计，避免影响消费分析。

### 4.3 账本管理

账本用于区分不同场景的账单。

示例账本：

- 日常账本
- 旅行账本
- 学习账本
- 家庭账本
- 情侣账本

账本功能：

- 新增账本
- 编辑账本名称
- 设置账本图标
- 设置账本颜色
- 设置默认账本
- 归档账本

删除账本需要谨慎处理。如果账本下已有账单，第一版建议不允许直接删除，而是提供归档功能。

### 4.4 预算功能

预算用于帮助用户控制支出。

预算类型：

- 月总预算
- 分类预算

预算页面需要展示：

- 当前月份
- 当前账本
- 总预算进度
- 分类预算列表
- 已用金额
- 剩余金额
- 超支状态

预算状态规则：

```text
使用率 < 80%：正常
80% <= 使用率 < 100%：提醒
使用率 >= 100%：超支
```

### 4.5 消费分析

消费分析分为本地分析和 AI 分析。

本地分析包括：

- 支出分类占比
- 每日支出趋势
- 月度收支对比
- 预算使用率
- 本月与上月对比

AI 分析包括：

- 本月消费总结
- 主要消费方向
- 异常消费提醒
- 预算执行情况
- 下月节省建议
- 奶龙风格鼓励语

AI 分析不应该替代本地图表，而应该作为本地统计之后的文字解释层。

- 关于应用

字段建议：

```text

```

### 6.4 Transaction 账单表

字段建议：

```text

categoryId

```

金额建议使用“分”为单位的整数存储。

示例：

```text

```

### 6.5 Budget 预算表

字段建议：

```text

```

当 categoryId 为空时，表示月总预算；当 categoryId 不为空时，表示分类预算。

### 6.6 AiAnalysisReport AI 分析报告表

字段建议：

```text

period

createdAt
updatedAt
```

该表用于缓存 AI 分析结果，避免用户每次打开分析页都重复调用 DeepSeek API。

## 7. DeepSeek AI 分析方案

正式版本不建议 Android App 直接调用 DeepSeek API，因为 API Key 放在客户端存在被反编译和盗用的风险。

推荐调用链：

```text
Android App
→ 自己的后端服务
→ DeepSeek API
→ 后端返回分析结果
→ App 展示并缓存报告
```

App 不应该把完整账单明细直接发送给 AI，而应该先在本地生成统计摘要，再发送给后端。

示例摘要：

```text
本月收入：5200 元
本月支出：3560 元
本月结余：1640 元
餐饮：1320 元，占比 37%
交通：420 元，占比 12%
娱乐：680 元，占比 19%
预算使用率：82%
相比上月，餐饮增加 23%
```

AI 输出格式建议固定为：

```text
本月总结
主要消费方向
异常变化提醒
预算执行情况
节省建议
奶龙鼓励语
```

AI 分析页面需要支持以下状态：

```text
未生成
生成中
生成成功
生成失败
```

生成失败时不能影响本地图表和基础记账功能。

## 8. 技术可行性分析

项目技术上可行，整体难度属于中等。

主要可行性结论：

- Android 端可以使用 Kotlin + Jetpack Compose 实现现代移动端界面。
- 本地数据可以使用 Room / SQLite 保存。
- 首页统计、预算统计、消费图表都可以基于本地数据库完成。
- DeepSeek API 可以作为 AI 分析能力来源。
- 后端代理可以保护 API Key，并控制调用频率和成本。

主要风险：

### 8.1 API Key 安全风险

如果把 DeepSeek API Key 写进 Android 客户端，发布后可能被反编译获取。

解决方案：

- 使用后端代理 DeepSeek API。
- API Key 只保存在后端环境变量中。
- Android 端只请求自己的后端接口。

### 8.2 用户隐私风险

账单数据属于敏感个人数据。

解决方案：

- AI 分析只上传统计摘要，不上传完整原始账单。
- 不上传具体商家、备注、精确时间等敏感字段。
- 在设置中明确提示用户 AI 分析会使用脱敏后的统计数据。

### 8.3 AI 服务不可用风险

网络异常、接口限流、余额不足都可能导致 AI 分析失败。

解决方案：

- 本地分析功能必须独立可用。
- AI 分析失败时展示可重试状态。
- 已生成报告要缓存，减少重复请求。

## 9. 技术架构建议

Android 端建议：

```text
Kotlin
Jetpack Compose
Room / SQLite
ViewModel + StateFlow
Retrofit / OkHttp
WorkManager
```

后端建议：

```text
FastAPI 或 Node.js
DeepSeek API 代理
API Key 保护
请求限流
错误日志
```

整体架构：

```text
Android App
  ├─ UI 层：页面、组件、用户交互
  ├─ ViewModel 层：页面状态、事件处理
  ├─ Domain 层：记账、预算、统计、AI 分析业务逻辑
  ├─ Data 层：Room 数据库、Repository、网络接口
  └─ 后端服务：DeepSeek API 代理、限流、日志
```

## 10. 模块架构分析

第一版可以拆成以下业务模块：

```text
记账模块
账本模块
预算模块
分析模块
AI 模块
设置模块
```

### 10.1 记账模块

职责：

- 新增账单
- 编辑账单
- 删除账单
- 查询账单列表
- 按日期、分类、账本筛选账单

依赖数据：

- Transaction
- Category
- Account
- Ledger

关键规则：

- 支出计入月支出。
- 收入计入月收入。
- 转账不计入收入和支出。
- 删除账单建议使用软删除，便于后续恢复或同步。

### 10.2 账本模块

职责：

- 创建账本
- 编辑账本
- 设置默认账本
- 切换当前账本
- 归档账本

关键规则：

- App 首次启动时自动创建默认账本。
- 每条账单必须属于一个账本。
- 已有账单的账本不建议直接删除。

### 10.3 预算模块

职责：

- 设置月总预算
- 设置分类预算
- 计算预算使用率
- 判断预算状态

依赖数据：

- Budget
- Transaction
- Category
- Ledger

关键规则：

- 总预算统计当前账本当月所有支出。
- 分类预算只统计对应分类的当月支出。
- 预算金额使用整数分存储。

### 10.4 分析模块

职责：

- 生成分类占比
- 生成支出趋势
- 生成收入支出对比
- 生成预算使用统计
- 生成本月与上月对比

依赖数据：

- Transaction
- Budget
- Category
- Ledger

关键规则：

- 分析数据优先在本地计算。
- 图表数据和 AI 摘要可以共用同一套统计结果。
- 分析维度至少支持按账本和月份筛选。

### 10.5 AI 模块

职责：

- 生成 AI 分析摘要
- 请求后端 AI 分析接口
- 展示 AI 报告
- 缓存 AI 报告
- 处理失败和重试

依赖数据：

- AiAnalysisReport
- 本地统计摘要
- 后端 AI 接口

关键规则：

- 不上传完整原始账单。
- 不上传备注等敏感字段。
- 同一账本、同一月份已有报告时优先读取缓存。
- 用户主动刷新时才重新生成。

### 10.6 设置模块

职责：

- 分类管理
- 账户管理
- 数据导出
- 主题设置
- AI 设置
- 关于应用

第一版中，设置模块只需要覆盖基础管理能力，不建议加入过多复杂配置。

## 14. 软件工程化设计原则

后续开发需要遵循以下软件工程原则，保证项目不是简单堆功能，而是具备可维护、可测试、可扩展的结构。

### 14.1 分层清晰

Android 端建议采用分层结构：

```text
UI 层
→ ViewModel 层
→ Domain / UseCase 层
→ Repository 层
→ LocalDataSource / RemoteDataSource 层
```

各层职责：

- UI 层只负责页面展示和用户交互。
- ViewModel 层负责页面状态管理和事件分发。
- Domain / UseCase 层负责业务规则，例如预算计算、账单统计、AI 摘要生成。
- Repository 层负责统一数据访问，对上层屏蔽本地数据库和网络来源。
- DataSource 层负责具体的 Room 数据库操作和后端接口调用。

### 14.2 高内聚、低耦合

每个模块只关注自己的业务职责。

示例：

- 记账模块不直接处理 AI 请求。
- AI 模块不直接查询原始账单表，而是接收分析模块生成的统计摘要。
- 预算模块只暴露预算状态和预算使用率，不关心页面如何展示颜色。

### 14.3 本地优先

基础记账、预算、账本、图表分析必须在无网络状态下可用。

AI 分析属于增强功能，不能成为核心功能的前置条件。

### 14.4 隐私优先

用户账单数据属于敏感数据。AI 分析只能使用脱敏后的统计摘要，不上传用户完整明细、备注、精确消费时间等敏感信息。

### 14.5 可扩展优先

第一版虽然不做云同步、共享账本、小票识别，但数据结构和模块边界要预留扩展空间。

例如：

- Transaction 保留 createdAt、updatedAt、deletedAt，方便以后同步。
- Ledger 保留归档能力，避免直接删除数据。
- AiAnalysisReport 独立存储，方便以后支持周报、月报、年报。

| NFR-002 | 可用性 | 记一笔的操作路径要短 | 常用支出 5 步内完成 |
| NFR-003 | 可靠性 | AI 失败不能影响本地记账 | 断网时记账、预算、图表可用 |
| NFR-004 | 安全性 | DeepSeek API Key 不得出现在 Android 客户端 | Key 只存放在后端环境变量 |
| NFR-005 | 隐私 | AI 分析不得上传完整账单明细 | 只上传统计摘要 |
| NFR-006 | 可维护性 | 模块职责边界明确 | UI 不直接访问数据库或网络 |
| NFR-007 | 可测试性 | 核心统计逻辑可单元测试 | 预算、统计、摘要生成有测试用例 |
| NFR-008 | 可扩展性 | 后续可支持云同步和共享账本 | 数据表保留时间戳和软删除字段 |

### 16.3 UC-004：生成 AI 月报详细流程

主成功流程：

```text
1. 用户进入分析页。
2. 用户选择账本和月份。
3. App 查询本地数据库，生成统计摘要。
4. App 检查本地是否已有同账本同月份 AI 报告。
5. 如果存在缓存，优先展示缓存。
6. 如果用户点击重新生成，App 将摘要发送到后端。
7. 后端校验请求并调用 DeepSeek API。
8. DeepSeek 返回分析内容。
9. 后端将结果返回给 App。
10. App 保存 AiAnalysisReport 并展示报告。
```

异常流程：

```text

后端失败：保留页面，不影响账单和预算功能。
DeepSeek 超时：后端返回明确错误码，App 展示重试入口。
无足够账单数据：AI 区域提示“本月记录较少，暂不生成完整报告”。
```

## 19. DeepSeek Prompt 设计初稿

AI 分析的 Prompt 应该稳定、结构化、可控，避免生成过长或偏离记账场景。

### 19.1 System Prompt

```text
你是“奶龙记账”的消费分析助手。你的任务是根据用户的月度收支统计摘要，生成简洁、温和、实用的中文消费分析报告。

要求：
1. 不要编造输入中不存在的具体消费记录。
2. 不要输出医疗、投资、法律等专业建议。
3. 不要批评用户，语气要友好、鼓励、低压力。
4. 优先指出消费结构、预算风险和可执行的节省建议。
5. 输出必须是 JSON，不要输出 Markdown。
```

### 19.2 User Prompt

```text
请根据以下月度消费摘要生成消费分析报告：

账本：{{ledgerName}}
月份：{{period}}
收入：{{income}}
支出：{{expense}}
结余：{{balance}}
预算：{{budget}}
预算使用率：{{budgetUsageRate}}
分类支出：{{categoryExpenses}}
上月对比：{{monthComparison}}

请输出以下 JSON 字段：
summary
mainCategories
alerts
budgetComment
suggestions
encouragement
```

### 19.3 输出约束

AI 输出必须能被后端解析。后端应校验字段完整性，如果 DeepSeek 返回非 JSON 内容，后端需要做失败处理或重试一次。

## 24. 页面原型级交互设计总则

页面设计需要围绕“快速记账、清楚预算、看懂分析”三个目标展开。

### 24.1 页面设计原则

- 常用操作优先：记账、查看预算、查看本月支出必须容易找到。
- 数据优先展示：金额、预算、趋势等核心信息应比装饰元素更突出。
- 状态明确：加载、空数据、失败、超支、已缓存等状态必须清楚。
- 交互可恢复：删除、退出编辑、AI 分析失败等场景要提供确认或重试。
- 移动端友好：所有主要操作应适合单手完成，底部按钮和输入区域要便于点击。

### 24.2 全局导航结构

底部导航：

```text
首页 / 记账 / 预算 / 分析 / 我的
```

推荐导航规则：

- App 启动后默认进入首页。
- 底部导航常驻，新增记账页面可以作为中间主入口。
- 从账单详情、预算编辑、AI 报告详情等二级页面返回时，应回到进入前的一级页面。
- 当前账本是全局上下文，首页、预算、分析默认使用同一个当前账本。

### 24.3 全局页面状态

所有列表类页面应至少支持：

```text
Loading     加载中
Success     成功展示数据
Empty       无数据
Error       加载失败
Refreshing  下拉刷新或重新计算中
```

表单类页面应至少支持：

```text
Editing     正在编辑
Submitting  正在保存
SubmitError 保存失败
SubmitDone  保存成功
```

## 25. 首页原型级说明

### 25.1 页面目标

首页承担“财务仪表盘”的角色。用户进入 App 后，应立刻知道当前账本本月的整体状态。

### 25.2 页面组件

建议组件从上到下排列：

```text
顶部区域
  当前账本切换
  当前月份

核心统计区域
  本月支出
  本月收入
  本月结余

  总预算进度条

最近账单区域
  最近 5-10 条账单
  查看全部入口

快速操作区域
  记一笔按钮
```

### 25.3 主要按钮和交互

| 组件 | 操作 | 结果 |
| --- | --- | --- |
| 当前账本 | 点击 | 打开账本切换弹窗 |
| 当前月份 | 点击 | 打开月份选择器 |
| 记一笔按钮 | 点击 | 跳转到记账页 |

### 25.4 空状态

当当前月份没有账单时：

```text

最近账单区域显示“还没有账单，记下第一笔吧”
```

### 25.5 异常状态

### 25.6 首页验收标准

- 切换账本后，统计数据同步切换。
- 切换月份后，支出、收入、预算、最近账单同步切换。
- 新增账单后返回首页，统计立即刷新。
- 删除账单后返回首页，统计立即刷新。

## 26. 记账页原型级说明

### 26.1 页面目标

记账页目标是让用户用尽可能少的步骤完成一笔账单记录。

### 26.2 页面组件

```text
类型切换
  支出 / 收入 / 转账

金额输入
  数字键盘
  金额显示

分类选择
  分类图标网格

账户选择
  支付账户 / 收入账户 / 转出账户 / 转入账户

账本选择
  默认当前账本

日期选择
  默认今天

备注输入
  可选

保存按钮
```

### 26.3 支出记账流程

```text
1. 用户进入记账页。
2. 系统默认选中“支出”。
3. 用户输入金额。
4. 用户选择支出分类。
5. 用户选择支付账户。
6. 系统默认使用当前账本和今天日期。
7. 用户点击保存。
8. 系统校验表单。
9. 保存成功后返回首页或停留并提示成功。
```

### 26.4 收入记账流程

```text
1. 用户切换到“收入”。
2. 输入金额。
3. 选择收入分类。
4. 选择收款账户。
5. 保存。
```

### 26.5 转账流程

```text
1. 用户切换到“转账”。
2. 输入金额。
3. 选择转出账户。
4. 选择转入账户。
5. 保存。
```

转账不需要分类，且不参与收入、支出、预算统计。

### 26.6 表单校验

| 字段 | 校验规则 | 错误提示 |
| --- | --- | --- |
| 金额 | 必须大于 0 | 请输入有效金额 |
| 分类 | 支出和收入必须选择 | 请选择分类 |
| 账户 | 必须选择 | 请选择账户 |
| 转入账户 | 转账时必须选择，且不能等于转出账户 | 请选择不同的转入账户 |
| 账本 | 必须存在 | 请选择账本 |
| 日期 | 不能为空 | 请选择日期 |

### 26.7 保存后的行为

保存成功可以有两种策略：

```text
策略 A：返回首页，适合新手用户。
策略 B：清空表单并停留，适合连续记账。
```

第一版建议采用策略 A，并在后续设置中提供“连续记账模式”。

### 26.8 记账页验收标准

- 支出、收入、转账三种类型可正确保存。
- 金额不能为 0 或负数。
- 转账不会计入月收入或月支出。
- 保存失败时不会丢失用户已输入内容。

### 27.2 页面组件

```text
  当前账本
  当前月份
```

```text

2. 点击总预算区域的编辑按钮。
3. 输入预算金额。
```

```text

3. 输入预算金额。
4. 设置提醒阈值。
5. 保存。
```

### 27.5 预算状态

| 状态 | 条件 | 展示建议 |
| --- | --- | --- |
| 正常 | 使用率 < 80% | 普通提示 |
| 接近超支 | 80% <= 使用率 < 100% | 提醒色和提示文案 |

### 27.6 空状态

未设置预算时：

- 设置总预算后，首页和预算页都能显示剩余预算。
- 分类预算只统计该分类支出。
- 修改或删除账单后预算进度同步变化。
- 超过提醒阈值和超过 100% 时状态展示正确。

## 28. 分析页原型级说明

### 28.1 页面目标

### 28.4 AI 分析流程

```text
1. 用户进入分析页。
2. App 查询当前账本和月份的统计摘要。
3. App 检查是否已有缓存报告。

6. 用户点击生成。
7. App 请求后端接口。
8. 后端调用 DeepSeek。
9. App 保存报告并展示。
```

### 28.5 AI 状态

| 状态 | 展示内容 | 用户操作 |
| --- | --- | --- |
| 未生成 | 说明可生成月度分析 | 点击生成 |
| 生成中 | 加载状态 | 不允许重复点击 |
| 已生成 | 展示报告内容 | 可重新生成 |
| 生成失败 | 错误提示 | 可重试 |
| 数据不足 | 提示记录较少 | 引导继续记账 |

### 28.6 AI 报告展示结构

```text

主要消费方向

预算执行情况
节省建议
奶龙鼓励语
```

### 28.7 分析页验收标准

- 无网络时，本地图表仍可展示。
- AI 失败不会影响本地图表。
- 同账本同月份已有报告时可读取缓存。
- 重新生成报告时会覆盖或生成新版本，需要记录生成时间。

## 34. Room 数据库设计

Android 本地数据库建议使用 Room，底层仍是 SQLite。Room 的优势是类型安全、DAO 结构清晰、适合 MVVM 架构。

### 34.1 数据库存储原则

- 金额统一使用 Long 类型，单位为“分”。
- 时间统一使用 Long 时间戳或 ISO 字符串，编码时二选一并保持一致。
- 删除优先使用软删除字段 deletedAt，避免误删和方便后续同步。
- 列表查询默认排除 deletedAt 不为空的数据。
- 所有表预留 createdAt 和 updatedAt。
- 主键建议使用 String UUID 或 Long 自增。若未来考虑云同步，更推荐 String UUID。

第一版推荐：

```text
id 使用 String UUID
amount 使用 Long，单位分
createdAt / updatedAt / deletedAt 使用 Long 时间戳
period 使用 String，例如 2026-06
```

### 34.2 LedgerEntity

用途：保存账本信息。

字段：

```text
id: String
name: String
icon: String
color: String
isDefault: Boolean
isArchived: Boolean
createdAt: Long
updatedAt: Long
deletedAt: Long?
```

约束：

- name 不能为空。
- 同一时间只能有一个默认账本。
- isArchived 为 true 的账本不在首页切换列表中默认展示。
- deletedAt 不为空时视为已删除。

### 34.3 CategoryEntity

用途：保存支出和收入分类。

字段：

```text
id: String
name: String
type: String              expense / income
icon: String
color: String
sortOrder: Int
isSystem: Boolean
isHidden: Boolean
createdAt: Long
updatedAt: Long
deletedAt: Long?
```

约束：

- type 只能是 expense 或 income。
- 系统默认分类不建议物理删除，可用 isHidden 隐藏。
- sortOrder 用于控制分类网格顺序。

### 34.4 AccountEntity

用途：保存账户信息。

字段：

```text
id: String
name: String
type: String              cash / bank / alipay / wechat / credit / other
initialBalance: Long
icon: String
color: String
isDefault: Boolean
isArchived: Boolean
createdAt: Long
updatedAt: Long
deletedAt: Long?
```

第一版说明：

- initialBalance 可以先保留但不做复杂资产计算。
- 若后续做账户余额，可通过 initialBalance + 账单流水计算。

### 34.5 TransactionEntity

用途：保存账单记录，是核心表。

字段：

```text
id: String
ledgerId: String
type: String              expense / income / transfer
amount: Long
categoryId: String?
accountId: String
targetAccountId: String?
date: Long
note: String
createdAt: Long
updatedAt: Long
deletedAt: Long?
```

约束：

- amount 必须大于 0。
- expense 和 income 必须有 categoryId。
- transfer 的 targetAccountId 不能为空。
- transfer 的 accountId 和 targetAccountId 不能相同。
- 所有统计默认排除 deletedAt 不为空的数据。

统计规则：

```text
expense 计入支出和预算。
income 计入收入。
transfer 不计入收入、支出、预算。
```

### 34.6 BudgetEntity

用途：保存月预算和分类预算。

字段：

```text
id: String
ledgerId: String
categoryId: String?
period: String            例如 2026-06
amount: Long
alertEnabled: Boolean
alertThreshold: Int       默认 80
createdAt: Long
updatedAt: Long
deletedAt: Long?
```

约束：

- categoryId 为空表示月总预算。
- categoryId 不为空表示分类预算。
- 同一 ledgerId + period + categoryId 只能有一条有效预算。
- amount 必须大于 0。
- alertThreshold 建议范围为 1 到 100。

### 34.7 AiAnalysisReportEntity

用途：缓存 AI 分析报告。

字段：

```text
id: String
ledgerId: String
period: String
summaryJson: String
reportJson: String
modelName: String
status: String            generated / failed
generatedAt: Long
createdAt: Long
updatedAt: Long
deletedAt: Long?
```

说明：

- summaryJson 保存发送给后端的脱敏统计摘要。
- reportJson 保存后端返回的结构化报告。
- status 可支持失败缓存，但第一版也可以只保存成功报告。

### 34.8 AppSettingEntity

用途：保存本地设置。

字段：

```text
key: String
value: String
updatedAt: Long
```

可保存：

```text
currentLedgerId
themeMode
defaultAccountId
continuousAccountingEnabled
aiPrivacyAccepted
```

## 39. 业务规则汇总

### 39.1 金额规则

- 输入金额必须大于 0。
- 金额统一以分为单位存储。
- UI 展示时转换为元。
- 计算时不得使用浮点数保存金额。

### 39.2 账单规则

- 支出影响支出统计和预算。
- 收入影响收入统计。
- 转账只影响账户流水，不影响收入和支出。
- 删除账单后相关统计必须刷新。

### 39.3 账本规则

- 每条账单必须属于一个账本。
- 必须始终存在一个默认账本。
- 已有关联账单的账本不建议直接删除。
- 归档账本不出现在常用切换列表，但历史数据仍可查看。

### 39.4 分类规则

- 支出分类和收入分类分开。
- 系统分类可隐藏，不建议删除。
- 自定义分类可编辑。
- 已被账单使用的分类不建议直接删除。

### 39.5 预算规则

- 预算按账本和月份生效。
- 月总预算统计所有支出。
- 分类预算只统计指定分类支出。
- 转账不参与预算。
- 超支金额 = 已用金额 - 预算金额。

### 39.6 AI 分析规则

- AI 分析只使用脱敏统计摘要。
- AI 报告按账本和月份缓存。

## 42. DeepSeek 接入详细设计

DeepSeek API 使用 OpenAI 兼容格式。编码时建议使用官方文档确认最新 base_url、模型名称、参数和价格。

当前设计依据：

- base_url：`https://api.deepseek.com`
- Chat Completions 路径：`/chat/completions`
- JSON 输出：请求中使用 `response_format: {"type":"json_object"}`，并在 prompt 中明确要求输出 JSON。

### 42.1 请求参数建议

```json
{
  "model": "以官方当前推荐模型为准",
  "messages": [
    {
      "role": "system",
      "content": "系统提示词"
    },
    {
      "role": "user",
      "content": "用户消费摘要"
    }
  ],
  "temperature": 0.4,
  "response_format": {
    "type": "json_object"
  }
}
```

参数说明：

- temperature 不宜过高，避免报告不稳定。
- 必须要求 JSON 输出，方便后端解析。
- max_tokens 可在编码时根据报告长度控制。

### 42.2 Prompt 约束

System Prompt：

```text
你是“奶龙记账”的消费分析助手。你需要根据用户提供的月度收支统计摘要，生成简洁、温和、实用的中文消费分析报告。

要求：
1. 只能基于输入摘要分析，不要编造具体账单。
2. 不输出医疗、法律、投资等专业建议。
3. 语气友好，不批评用户。
4. 建议必须具体、可执行。
5. 输出必须是 JSON 对象，不要输出 Markdown。
6. JSON 字段必须包含 summary、mainCategories、alerts、budgetComment、suggestions、encouragement。
```

User Prompt 模板：

```text
请根据以下脱敏月度消费摘要生成消费分析报告。

账本：{{ledgerName}}
月份：{{period}}
币种：{{currency}}
收入：{{income}}
支出：{{expense}}
结余：{{balance}}
预算：{{budget}}
预算使用率：{{budgetUsageRate}}
账单数量：{{transactionCount}}
分类支出：{{categoryExpenses}}
日均支出：{{dailyAverageExpense}}
最高支出分类：{{topExpenseCategoryName}}

请输出 JSON：
{
  "summary": "string",
  "mainCategories": ["string"],
  "alerts": ["string"],
  "budgetComment": "string",
  "suggestions": ["string"],
  "encouragement": "string"
}
```

### 42.3 后端校验规则

后端收到 DeepSeek 返回后需要校验：

- 是否为合法 JSON。
- 是否包含全部必填字段。
- mainCategories、alerts、suggestions 是否为数组。
- 字符串长度是否在合理范围。
- 不允许返回空报告。

校验失败时：

```text
可以自动重试一次。
重试失败后返回 AI_502_PROVIDER_FAILED。
```

### 44.2 色彩建议

- 说明文字使用较小字号。
- 金额小数保持两位。
- 支出和收入颜色区分要稳定。

### 44.4 奶龙元素使用边界

- 空状态插画。

不建议用于：

- 遮挡金额。

```text

AI 报告卡片
```

## 45. 安全与隐私设计

### 45.1 API Key 安全

禁止：

```text
Android 客户端内置 DeepSeek API Key。
将 API Key 写入 Git 仓库。
在日志中打印 API Key。
```

必须：

```text
API Key 只保存在后端环境变量。
后端调用 DeepSeek。
客户端只请求自己的后端。
```

### 45.2 用户隐私

AI 分析只上传：

```text
总收入
总支出
预算使用率
分类汇总
趋势摘要
环比变化
```

AI 分析不上传：

```text
单笔账单明细
备注
具体商家
精确消费时间
账户名称中的敏感信息
```

### 45.3 日志安全

后端日志可以记录：

```text
requestId
接口耗时
错误码
模型名称
```

后端日志不应记录：

```text
完整消费摘要
AI Key
用户敏感备注
```
